# Docs Contribution Model

Docs-only changes are encouraged and should be low-friction.

## Scope

- Clarifications and corrections
- IA/navigation improvements
- Onboarding and runbook updates
- API/reference examples for public surfaces

## Contribution rules

- Keep docs accurate to shipped behavior.
- Prefer concise, operator-first language.
- Link to canonical files and issues.
- Include validation or evidence where applicable.

## PR expectations

- Mark PR as docs-only in title or summary.
- Include before/after impact in one paragraph.
- Keep unrelated code changes out of docs-only PRs.
